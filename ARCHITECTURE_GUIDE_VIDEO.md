# ChatPulse 技术科普（视频脚本版）

## 开场：这个项目到底在干嘛

大多数 AI 聊天应用，角色都是没有状态的。你跟它说一句，它回一句，关掉页面什么都不记得了。下次再打开，它跟你又是一个陌生人。

ChatPulse 想做的事不一样。这里的 AI 角色是"活着"的——它会记住你之前说过的话，有自己的情绪起伏，你不找它的时候它自己去上班、吃饭、逛街，甚至会主动给你发消息来找你聊天。它有私聊、有群聊、有朋友圈、有一整个模拟城市让角色在里面生活。

听起来好像很简单对吧？但要让一个 AI 角色真的做到"记住你"和"像活的一样"，背后其实需要解决几个很棘手的技术问题。今天就来拆解一下这个项目里最核心的三块技术：缓存系统、RAG 检索增强生成、还有基于向量数据库的长期记忆。


## 缓存系统：别做重复劳动

先说缓存。缓存这个词听起来很技术，但本质上就是"备忘录"——算过的东西先存下来，下次要用直接拿，不用重新算。

每次角色要跟你说一句话，系统在后台其实做了很多事。它得把角色的人设、行为准则、世界观设定这些东西拼成一大段文字，然后把最近的聊天记录做格式转换，最后把这些东西一起扔给大语言模型，让它生成一条回复。

问题在哪呢？你跟角色连续聊了十几句，角色的人设变了吗？没有。世界观变了吗？也没有。但系统每次都要重新把这些东西拼一遍。这就像你每天上班都要重新画一张公司地图——明明路线没变过，画它干嘛。

ChatPulse 搭了三层缓存来解决这件事。

第一层叫 Prompt Block 缓存。它管的是角色人设、行为准则、情绪和身体状态描述这些"大块文本"。每次拼这些文本之前，系统会先对源数据算一个 SHA-256 哈希——你可以理解成给数据拍一张指纹照。如果这次的指纹跟上次一模一样，说明数据没变，直接拿上次的编译结果就行，不用重新拼了。

第二层叫 History Window 缓存，管的是聊天记录的格式转换。聊天记录是在不停变的，但变的只是最新那几条——前面的旧消息还是那些。所以系统用了一个滑动窗口的方式：你新发了两条消息，它只处理这两条新的，前面已经处理过的部分直接从旧缓存里继承过来。像活页笔记本一样，只翻新的页，不重抄旧的。

第三层是 LLM Response 缓存。这一层管的是 RAG 流程中间那些 LLM 调用的结果。后面会讲到，RAG 管线在生成回复之前要调好几次小模型做分析——话题识别、检索决策、查询改写。如果六小时之内输入完全一样，就直接拿上次的结果，不再花钱调一次 API。

这三层缓存叠在一起，直接的结果就是响应更快、token 消耗更少。项目实测下来，配合后面要讲的摘要压缩模式，prompt 长度可以缩短大约百分之四十到五十。


## RAG：让角色"回忆起来"

好，现在说 RAG。RAG 全称是 Retrieval-Augmented Generation，中文叫"检索增强生成"。这个名字很学术，但翻译成人话就三个字：先搜再答。

大语言模型有一个天生的限制——它的上下文窗口是有限的。你跟角色聊了一百条消息，窗口可能只装得下最近十几条。窗口之外的东西，模型根本看不见。那如果用户突然问"你还记得我上周跟你说的那件事吗？"，模型怎么办？只能瞎编。

RAG 就是为了解决这个问题。它在模型生成回复之前，先去记忆库里搜索跟当前对话相关的旧信息，搜到了就塞进 prompt 里。这样模型就"看见"了那些上下文窗口之外的旧记忆，回答就不再是瞎猜了。

但 ChatPulse 的 RAG 不是简单地每次都搜。它搭了一个四阶段的智能管线。

第一阶段叫话题提取。用户发了一条消息之后，系统不急着搜，而是先用一个轻量级的 LLM 调用来分析这条消息：用户在聊什么？可能涉及哪些长期记忆话题？比如说用户问了一句"最近面试怎么样了"，系统可能会推断出"用户近况"、"求职"、"工作"这几个潜在话题。它特别偏好跟用户个人信息、当前生活弧线、以及角色跟用户关系相关的话题。

第二阶段叫检索决策。有了话题之后，系统接着问另一个问题：现在真的需要去翻旧记忆吗？有时候最近的聊天记录就已经够了——比如用户只是说了句"晚安"，翻记忆毫无意义。只有当旧记忆能让回答更准确、更连贯、更个人化的时候，系统才决定触发搜索。它会输出两种结论：ENOUGH_CONTEXT 意思是不用搜了，SEARCH_MEMORY 意思是需要搜。这一步避免了"每句话都去翻记忆"带来的延迟和 token 浪费。

第三阶段叫查询改写。前一步给出的搜索关键词通常比较粗糙，比如就是"用户近况"几个字。这一步把它改写成一个结构化的搜索请求——包含多条搜索短语、记忆类型过滤器、还有结果数量上限。而且系统会在这一步叠加硬约束：如果检测到用户在问"你了解我吗"这类问题，就强制加入"用户画像"这个过滤器，确保不会漏掉。

第四阶段就是真正的向量检索了。拿着改写后的查询，去 Qdrant 向量数据库里找语义最相近的记忆条目。找到的记忆会被格式化成一段文字，注入到系统提示词里，让角色"回忆起来"。


## 记忆系统：角色的大脑怎么运作

最后说记忆系统。这是让角色真正"记住你"的核心。

ChatPulse 用了两个数据库来协作管理记忆：SQLite 和 Qdrant。

SQLite 存的是记忆的完整数据——事件、时间、人物、情绪、重要度、跟用户的关系分类，全都有字段对应。它擅长精确查询，比如"给我这个角色的所有核心记忆"或者"按重要度排序"。你可以把它理解成图书馆的书架和目录卡，什么书在哪个架子上，分类清清楚楚。

Qdrant 是一个向量数据库，它存的不是文字本身，而是每条记忆对应的一个高维数字向量——一个 1024 维的数组。这个向量是怎么来的呢？系统在本地跑了一个叫 BGE-M3 的嵌入模型，把记忆的文字内容转成一串数字。这串数字可以理解为这段文字的"语义指纹"——意思相近的文字，指纹也会相近。当用户说了一句话，系统把这句话也转成向量，然后在 Qdrant 里用余弦相似度搜索最相近的向量。这就是所谓的"语义搜索"——不是按关键词匹配，而是按意思匹配。你可以把它理解成图书馆里的"以意搜书"搜索引擎：你描述一个大概意思，它帮你找出最相关的书，哪怕你用的词跟书名完全不一样。

重点是，嵌入模型跑在本地，不依赖外部 API。这意味着不花额外的钱，也不受第三方服务的速度限制。

两个数据库各有分工：Qdrant 负责"找到最相关的记忆"，SQLite 负责"把记忆的完整信息取出来"。搜索的时候先去 Qdrant 按语义找，找到对应的 ID 之后再去 SQLite 拿完整数据。而且如果 Qdrant 没装——比如用户没跑 Docker——系统会自动降级到本地的 Vectra 索引加上 SQLite 的词汇匹配，功能照样能用，只是搜索精度稍差一点。

再说记忆是怎么产生和管理的。每轮对话结束后，系统用 LLM 从对话内容里提取"值得记住的事情"。注意不是逐字存，是提炼——比如一段五十条消息的聊天，系统可能只提取出三条核心记忆：用户分享了什么、角色跟用户之间发生了什么情绪变化、用户提到了什么重要的事。

每条记忆被保存的时候会被打上两个分类标签。一个叫 memory_focus——这条记忆是关于"用户个人信息"、"用户当前近况"、"角色和用户的关系"还是"一般事件"。另一个叫 memory_tier——这条记忆是"核心不变的"（比如用户的名字、学校）、"活跃变化的"（比如用户最近在找工作）还是"背景氛围级别的"（比如一个普通的闲聊片段）。这些标签在检索时用来做精确过滤，让搜索结果更相关。

时间久了记忆会越来越多，系统有两个机制来应对。一个是日级聚合：每天自动把零散的小记忆合并成日级摘要，减少碎片。另一个叫溢出消化：聊天记录超过上下文窗口的上限之后，超出部分的旧消息会被提炼成长期记忆再标记为"已消化"，确保信息不丢失。

检索记忆的时候，系统不是简单地"最相似的排最前面"就完事了。每条候选记忆会经过一套打分机制：向量相似度是底分，但 core 级别和 user_profile 类型的记忆有额外加分；如果记忆文本里直接包含了搜索关键词，再加一波词汇匹配分；中英文别名能互相桥接，比如用户搜"工作"也能匹配含"work"的记忆；甚至还有矛盾惩罚——如果一条记忆跟其他记忆信息矛盾，分数会被压低。最后同一组内容去重，只留最高分的那条。


## 对话摘要压缩：Digest + Tail

聊多了 prompt 就会越来越长，怎么办？ChatPulse 用了一个叫 Digest + Tail 的策略。

假设你跟角色聊了六十条消息。系统不会把这六十条全塞进 prompt 里——塞不下也太贵了。它会把旧的大量消息用 LLM 压缩成一段简短的对话摘要，只保留"前面大致聊了什么"的信息。然后把最近的几条消息保持原文不动。

结果就是 AI 既知道"前面聊了什么大致内容"——通过摘要，又能精确看到"刚刚说了什么"——通过尾部原文。prompt 长度砍了一大半，但信息几乎没有丢失。


## 统一上下文构建器：把所有信息粘在一起

最后简单提一下统一上下文构建器。不管角色是在私聊、群聊还是城市模拟里，生成回复之前都会经过同一个模块，拿到一份统一的"世界状态简报"。

这份简报包含当前时间和对应的行为约束、角色的身体状态比如精力值和睡眠债和饱腹感、当前情绪和压力等级、城市生活场景——正在工作还是在休息还是饿了、钱包余额和体力百分比、嫉妒状态和被冷落的记录、对其他角色的好感和历史印象、朋友圈动态、群聊和私聊之间的跨场景信息注入、商业街的最近活动记录、以及 RAG 检索回来的长期记忆。

这些信息全部整合在一起，塞进系统提示词最前面。这就保证了角色在不同场景下表现是一致的。它在城市里打了一天工觉得累，回来私聊的时候也会带着那种疲惫感——而不是突然精力满满像什么都没发生过。


## 功能介绍：角色的社交世界

前面讲了底层技术，现在来聊聊用户真正能看到和玩到的功能。ChatPulse 不是一个单纯的对话框，它模拟了一整套社交生态——私聊、群聊、转账红包、朋友圈、还有一个让角色自己生活的商业街。


## 私聊：不只是一问一答

私聊是最核心的交互场景。界面长得像微信的聊天窗口，但底下跑的东西比普通聊天复杂得多。

首先，每个角色都有自己独立的 API 配置——用的什么模型、什么接口、什么密钥，都可以单独设。你可以让一个角色用 Claude，另一个用 GPT，还有一个用本地模型，它们各走各的通道。

其次，角色不是只有你找它它才说话。引擎里有一套主动消息机制。系统会根据角色的性格、跟你的关系、以及当前状态，按一个随机时间间隔给你主动发消息。角色如果在城市里遇到了什么事——比如打工太累了想找人聊天，或者在逛街的时候看到了什么好玩的——它就会自己给你发一条过来。这不是定时推送的模板消息，每一条都是 LLM 现场根据角色当前的状态和关系实时生成的。

私聊里还有一个转账功能。你可以给角色发钱。有意思的是，角色收到转账之后不是自动入账的——系统会调一次 LLM 让角色自己"决定"要不要收。它会根据自己的性格、跟你的好感度、还有转账的金额和备注，做出接受或者退还的判断。比如你给一个傲娇角色发了一笔转账写着"对不起"，角色可能看了一会儿之后把钱退给你，然后说一句"不稀罕你的钱"。也可能心软了一下收了下来但嘴上还是不承认。这个选择不是写死的逻辑，是模型现场推理的。

如果角色正在拉黑你的状态下你给它转账，它甚至有可能因为被打动而解除拉黑。退还的时候角色也不是无声退款，系统会结合最近的对话上下文和转账发出到退还之间过了多久，让角色说一句自然的退款理由。

除了转账，你还可以在私聊里给角色推荐名片——把另一个角色介绍给它认识，角色之间就多了一层社交关系。

所有私聊消息通过 WebSocket 实时推送，角色的打字思考状态、倒计时、情绪变化都会实时显示在界面上，不用手动刷新。


## 群聊：多角色同台

群聊是另一个主要场景。你可以建一个群，把好几个角色拉进来，然后看它们互动。

群聊的 AI 链式触发机制很有意思。你在群里发了一条消息之后，系统会判断哪些角色可能想回话。如果你 @ 了某个角色，它一定会回。如果你 @ 了全体成员，所有角色都会依次回一轮。即使你没有 @，系统也会根据每个角色的性格、跟你的关系、当前情绪和话题相关性来计算回话概率——外向的角色更容易插嘴，内向的可能潜水不动。

而且触发不只是一轮。一个角色在群里说了话之后，其他角色可能被这句话刺激到产生二次回应。比如角色 A 在群里酸了你一句，角色 B 可能跳出来替你说话，角色 A 又可能不爽地怼回去。这个链式反应是递归触发的，不过系统做了深度限制防止无限循环。

每个角色在群聊里看到的上下文是统一的——群里谁说了什么它都能看见。但同时，它也能"偷偷"带入一些私聊里的情绪。前面讲统一上下文构建器的时候提过，群聊语境下系统会把角色跟你的私聊内容作为"秘密上下文"注入。角色在群里不会直接说出你们私聊的内容，但私聊里的情绪、闹别扭、吃醋这些东西会自然地渗透到它在群里的表现——比如它可能在群里对你说话比较冲，或者故意对别的角色更热情来刺激你。

群聊同样有对话摘要压缩。群消息量通常更大，系统用跟私聊类似的 digest+tail 模式来控制上下文长度——旧消息被压缩成摘要，最近几条保持原文。


## 红包：群里的社交游戏

群聊里有红包功能，这是经济系统的一部分。你可以在群里发红包，有两种类型——普通红包是每份固定金额，拼手气红包是总金额随机分配。

红包发出来之后不是只有你能抢。群里的 AI 角色看到红包也会去抢。系统会触发一轮群聊 AI 链，让角色们对红包做出反应——有的角色可能兴冲冲地抢了然后感谢一句，有的可能嘀咕金额太小，有的可能因为性格矜持假装没看见但其实悄悄点了。

红包的领取记录是实时的。谁抢了多少、还剩几份，都通过 WebSocket 推送到界面上。所有人的钱包余额也是实时同步的——角色在城市里打工赚的钱、发红包花掉的钱、收到转账的钱，都归进同一个钱包体系。

角色用自己赚来的钱发红包是有上限的。引擎在行为准则里明确告诉角色：发转账的金额不能超过自己的钱包余额。角色如果想给你发一个五块二的红包道歉，但钱包里只有三块，它就只能发三块或者换个方式表达。


## 朋友圈：角色也发动态

朋友圈模拟的是微信朋友圈。角色可以主动发布动态——可能是它逛街看到什么了、工作累了吐槽一句、或者在想你了发一条矫情的。你可以点赞和评论，角色也可以互相点赞评论。

角色发朋友圈这件事不需要你触发。引擎的行为准则里给角色定义了一个 MOMENT 标签——角色在回话过程中如果觉得"这个事值得发个朋友圈"，它会在回复里附带一个 MOMENT 指令，后台自动提取发布。同理，角色如果想给别人的动态点赞或评论，也有对应的 MOMENT_LIKE 和 MOMENT_COMMENT 标签。

朋友圈的内容会被注入到角色的上下文里。统一上下文构建器在拼世界状态的时候，会把最近的朋友圈动态放进去——让角色知道"最近谁发了什么动态，谁给谁点了赞"。这意味着你在朋友圈发了一条动态之后，角色在私聊里可能会提到它——"你刚发的那条朋友圈是什么意思啊"。


## 商业街：角色的真实生活

商业街是整个项目最重的功能之一，代码量将近四千行。它本质上是一个城市生活模拟系统。

每个角色在城市里有一个当前位置。城市由多个区域组成——工厂、餐厅、便利店、公园、学校，用户也可以自定义区域。角色每隔一段时间会"行动"一次：去哪里、做什么、花了多少钱、赚了多少钱、精力变化了多少，这些都是系统自动调度的。

行动调度分几个层面。首先有一个被动生存 tick——每分钟角色的卡路里会自然消耗，饱腹感会下降，睡眠债会累积。饿到一定程度角色的状态会变成"饥饿"，困到一定程度会变成"补觉中"。然后有主动行动——系统用 LLM 给角色生成一天的日程表：几点去打工、几点吃饭、几点休息。日程表不是写死的，会根据角色当前的身体状态、钱包余额和性格动态调整。

最有意思的是行动和聊天之间的双向联动。角色在商业街的行动会直接影响它跟你聊天时的状态。比如角色正在打工，你给它发消息，它回你的时候会带着忙里偷闲的仓促感，说话可能更短更急，而且每多聊一句它的"工作分心值"就会上升。如果角色正在补觉被你消息吵醒了，它回你的时候语气会带着迷糊和不情愿，而且打断次数越多第二天醒来越累。

反过来也成立。你在私聊里跟角色说"你去吃点东西吧"，角色可能会在回话里附带一个 CITY_INTENT 标签，告诉系统"我要去餐厅"。然后系统就会真的让角色在城市里移动到餐厅区域并执行一次吃饭行为——钱包扣钱、饱腹感回升、精力恢复。

商业街还有随机事件系统。角色在逛街的时候可能遇到意外——捡到东西、被人搭话、看到打折、路上滑倒。这些事件发生后角色可能会主动给你发消息聊这件事。

角色之间在商业街也会社交碰撞。如果两个角色同时出现在同一个区域，系统会判断它们之间有没有关系、好感度够不够、当前状态适不适合社交，然后可能触发一段 AI 生成的社交场景——两个角色在公园偶遇聊了几句，或者在工厂里一起干活吐槽了一下今天的任务。

整个城市还有一个"市长 AI"——定期审视城市的经济状况和物价水平，做出宏观调整和事件投放。这是一个独立的 LLM 调用，专门负责城市层面的运营。


## 收尾

总结一下 ChatPulse 这套系统的核心思路。缓存系统让重复计算消失，三层缓存层层递进，节省了大量 token 和响应时间。RAG 管线让角色具备了"按需回忆"的能力——不是每句话都去翻记忆，而是聪明地判断什么时候需要翻、翻什么。双存储记忆架构让 SQLite 的结构化精确查询和 Qdrant 的语义模糊搜索互相配合，再加上多重评分加权和降级策略，让记忆检索既准确又鲁棒。统一上下文构建器则把时间、身体、情绪、生活、关系、记忆这些散落各处的信息整合成一份简报，保证角色在所有场景里都像同一个人。

这就是 ChatPulse 跟普通聊天机器人最本质的区别。它不是一个无状态的对话框，而是一个有记忆、有情绪、有生活的持续存在。
